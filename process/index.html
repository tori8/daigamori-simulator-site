DEVELOPMENT PROCESS

AIエージェントの分業で作る。アルファ版

このシミュレーターは、Anthropic社のAI開発ツール「Claude Code」を使い、役割分担したAIエージェントのチームが人間のディレクションのもとで開発しています。そのプロセスを紹介します。

Claude Code とは

人間が方針を決め、AIがコードを書き、検証し、記録する。

Claude Code は、対話で指示を受けてプログラムの実装・テスト・文書化までを行うAIコーディングエージェントです。本プロジェクトでは、1つのAIに全部を任せるのではなく、職人の工房のように役割を分けた複数のエージェントを立てて開発しています。

開発体制 — 6つの役割

図A: 開発体制。「作る人」と「検査する人」を分け、検査役は作り手の自己申告を信用しない決まり

各エージェントには「縄張り」(変更してよいファイルの範囲)が決められており、担当外のファイルには触れません。必要が生じたら棟梁経由で担当に依頼します。これにより、複数のAIが同時に作業しても衝突しません。

進め方の3つの特徴

1. 契約書ファースト(並行開発) — 描画エンジンと画面のように連携が必要な部分は、先に「契約書」(受け渡しの取り決めを書いた文書)を棟梁が作り、両者は互いのコードを見ずに契約書だけを頼りに同時進行します。会話の記憶ではなく文書で仕様を共有するのが原則です。

図B: 同時進行(並行開発)。フェーズ1では、この方式でエンジンと画面が一度の手戻りもなく結合できました

2. 批判的な監査 — 大きな節目ごとに監査役が成果物を検査します。監査役の決まりは「褒めるだけのレポートは書かない」「読んだだけで問題なしと書かない(必ず実行して確かめる)」。指摘は「重大 / 要修正 / 提案」に分類され、要修正以上は全件対応してから公開します。

実例: 3D機能の監査では、ブラウザを自動操作して2D↔3Dを20往復させメモリ漏れがないことを実測した上で、「説明文言が事実と異なるケースがある」等の指摘が出て、修正後にマージ(合流)しました。

3. 実物との照合(色検証) — 「それらしい色」で済ませないため、目利き役が実物の写真から色を数値(RGB/HSL)で採取し、シミュレーション出力と比較します。第1回検証では5つの見本レシピのうち3つで大きなずれが見つかり、「唐土×灰釉×還元は青緑が正しい」等の修正がデータに反映されました。

これまでの歩み

図C: タイムライン。新機能はブランチ(本番に影響しない作業用の分岐)で作り、監査を通してから合流します

数字で見る品質管理

6役割分業エージェント
2回節目の監査
15件監査指摘(全件対応)
0件重大指摘

すべての作業は「誰が・いつ・何をしたか」の作業ログとして記録され、監査レポート・色検証レポートも文書として残っています。AIの作業であっても、後から人間が経緯を追跡できることを重視しています。

人間とAIの役割分担

施主(人間)が決めること要件(何を作るか)・デザインの好み・公開/非公開の判断・価格などの運用方針・各節目のGOサイン
AIエージェントがやること実装・テスト・ブラウザでの動作検証・相互レビュー(監査/色検証)・文書化・デプロイ(公開作業)

たとえば「コードを非公開にしたい」「価格は頻繁に変わる」といった判断は施主が下し、その実現方法の設計と実装をAIが担いました。
